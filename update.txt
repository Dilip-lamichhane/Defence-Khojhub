1) Project Overview
Aim
KhojHub is a locationтАСbased business discovery platform that helps customers find nearby shops and lets shopkeepers register/manage their catalog. It combines a consumer map experience with a shopkeeper dashboard and an admin portal.

Core Capabilities
- MapтАСbased discovery (category filters, product search, distance filters)
- Shop registration with geo coordinates
- Shopkeeper dashboard (products, availability, hours)
- Admin portal (shop approval, moderation, analytics)
- Clerk authentication with Supabase RLS authorization

3) Backend Architecture (backend/)

3.1 server.js
Role
- Express app bootstrap, middleware registration, route mounting, error handling, Mongo connection, Supabase init.
Key behaviors
- CORS uses CORS_ORIGIN env or defaults to http://localhost:5173
- Routes:
	/api/v1/health
	/api/v1/auth
	/api/v1/clerk-auth
	/api/v1/shops
	/api/v1/products
	/api/v1/reviews
	/api/v1/categories
	/api/v1/supabase
	/api/v1/admin/enterprise (REAL Supabase)
	/api/v1/admin/dev (DUMMY Supabase)
- Geocode endpoints for reverse/search via Nominatim
- MongoDB connection (MONGODB_URI)
- Supabase config check via initializeSupabase()

3.2 config/supabase.js
Role
- Central Supabase client factory for REAL/DUMMY
- Returns anon/admin clients and validates env config
Key functions
- normalizeProject(value): DUMMY or REAL
- getActiveSupabaseProject(): reads ACTIVE_SUPABASE_PROJECT
- readProjectConfig(project): reads PROJECT_SUPABASE_* envs
- getSupabaseClient({ project }): anon client
- getSupabaseAdminClient({ project, purpose, allowDemo }): service-role client or anon fallback
- initializeSupabase(): validates active project keys
- isDemoSupabaseProject(): DUMMY mode check

3.3 middleware/auth.js
Role
- Clerk JWT verification for API routes
- Fetches Supabase user and resolves Mongo user
Inputs
- Authorization: Bearer <Clerk JWT>
Outputs
- req.auth { clerkId, email, role, supabaseUserId }
- req.user (Mongo user or null)

3.4 middleware/clerkSupabaseAuth.js
Role
- ClerkтАСonly authentication for clerkтАСsync flow
- Resolves Supabase user and role
Key functions
- requireClerkAuth(): verifies Clerk JWT
- requireSupabaseUser(): ensures user exists in Supabase
- requireRole(...roles): role gate
- getClerkEmail(clerkId): resolves email via Clerk API

3.5 middleware/rbac.js
Role
- Role and permission checks for admin/shop operations
Key functions
- authorize(...roles)
- authorizePermissions(...permissions)
- authorizeShopAction()
- isOwnerOrAdmin(resourceField)
- isShopOwner()

5) Data Flow & Logic Flow

Auth Flow
1. User signs in via Clerk.
2. AuthBootstrap requests Clerk token and calls backend /clerk-sync.
3. Backend upserts Supabase users table and Mongo user.
4. Frontend stores role and allows access to routes.

Shop Registration (Supabase)
1. User opens registration form on HomePage.
2. HighтАСaccuracy geolocation fills lat/lng (editable).
3. POST /api/v1/supabase/shops/register with x-supabase-project header.
4. Supabase shop row created with pending status.

Shop Approval (Admin)
1. Admin uses /admin portal.
2. Admin action updates Supabase shop status.
3. If approved, Mongo shop may be created (adminController logic).

Shop Discovery
1. Map UI calls Redux fetchSupabaseShops.
2. Backend returns shops from Supabase based on filters.
3. Map renders markers; open/closed glow.
4. Catalog sidebar fetches /supabase/shops/:id/products.

Shopkeeper Dashboard
1. Clerk token used to create authed Supabase client.
2. Loads shop by owner_id or email fallback.
3. Product CRUD operations directly in Supabase.

6) Supabase Schema & RLS
- supabase/schema-fix.sql
- Ensures users, shops, products tables and required columns
- owner_id and clerk_id are text (Clerk IDs)
- RLS policies use auth.jwt()->>'sub'
- Public read policy on shops/products for map
